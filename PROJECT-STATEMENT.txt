1 Overview
In this project you will design, implement and evaluate a sealed-bid double
auction (commonly called a dark auction) using Secure Multi-Party Com-
putation (MPC). The auction runs on top of the MP-SPDZ framework, which
provides several MPC protocols with different security models and performance
characteristics.
You will work in groups of 3 students. Each group member operates one MPC
party (node). The three parties jointly execute the auction program without any
single party learning the others’ private orders — only the final market outcome
(clearing price, traded volume, and individual fills) is revealed.
You are given the Docker infrastructure, networking configuration, input files
and helper scripts. You are NOT given any MPC program code. Your
task is to design and write the auction program (a .mpc file) from scratch, guided
by the algorithm described in Section 2.
The project spans 5 weeks, with one milestone per week.

2 Background: What Is a Dark Auction?
A dark auction (or dark pool call auction) is a market mechanism used in fi-
nancial exchanges where buy and sell orders are submitted secretly and matched
simultaneously at a single uniform price. Unlike a continuous exchange — where
every new order is immediately visible to all participants — a dark auction col-
lects orders during a submission window and only reveals the result after the
window closes.

2.1 Orders
There are two kinds of orders:

• Buy (bid) order: “I want to buy q units at up to p euros per unit.”
The bid price is the maximum the buyer is willing to pay.

• Sell (ask) order: “I want to sell q units at no less than p euros per unit.”
The ask price is the minimum the seller will accept.

An order is a tuple (price, quantity).

• 𝑆(𝑝) (the supply) is the total quantity sellers are willing to sell at price 𝑝

2.3 Clearing Price and Transacted Volume
The transacted volume at price 𝑝 is:

𝑉 (𝑝) = min(𝐷(𝑝), 𝑆(𝑝))
The goal is to find the price 𝑝∗ that maximises 𝑉 (𝑝). This price is called the
clearing price.
When multiple prices yield the same maximum volume (a plateau), a common
tie-breaking rule is to pick the midpoint of the plateau:

𝑝∗ = 𝑝low + 𝑝high

2.4 Pro-Rata Allocation
At the clearing price, one side of the market is typically over-subscribed (demand
> supply, or vice versa). The over-subscribed side must be rationed. Pro-rata
allocation distributes the available volume in proportion to each eligible order’s
quantity:

where 𝑞𝑖 is the quantity of eligible order 𝑖 and the sum is over all eligible orders

Example. Suppose the clearing price is 104.5 with volume 8. Eligible bids have
quantities 3, 5, 2 (total = 10). Pro-rata fills: ⌊3 × 8/10⌋ = 2, ⌊5 × 8/10⌋ = 4,
⌊2 × 8/10⌋ = 1, allocated = 7, residual = 1 given to the first order → fills = (3,
4, 1).

The protocols in Table above cover both dishonest-majority and honest-majority
backends available in MP-SPDZ. mascot-party.x implements MASCOT, a ma-
liciously secure protocol for arithmetic MPC over a prime field in the dishonest-
majority setting. Its main design goal is to keep the online phase efficient while
generating authenticated Beaver triples in an OT-based preprocessing phase
[1,2]. semi-party.x is the corresponding semi-honest OT-based variant; ac-
cording to the MP-SPDZ documentation, it can be seen as MASCOT/SPDZ2k
with the malicious-security machinery removed, leaving OT-based generation of
additively shared Beaver triples [1].
shamir-party.x is the honest-majority backend based on Shamir secret
sharing. At a high level, a secret is encoded as evaluations of a low-degree
polynomial, so any coalition below the reconstruction threshold learns nothing
about the secret; this is the classical foundation of BGW-style honest-majority

3



MPC [1,3,4]. replicated-ring-party.x is the semi-honest three-party honest-
majority backend over ℤ2𝑘 using replicated secret sharing. In MP-SPDZ, this
implementation follows the optimized replicated-sharing approach associated
with Araki et al., which is designed for very low communication and high
throughput in the 3-party setting [1,5].

3.2 Input Files
Each party’s private orders are stored in a human-readable file inputs/party{id}.txt:
# Party 0 — dark-auction orders
# Each line: bid_price bid_qty ask_price ask_qty
# A zero price means 'no order on that side'.

# ￿￿ BTC orders ￿￿
# bid_price bid_qty ask_price ask_qty

0 0 103 2
106 5 0 0
85 1 101 2

A conversion script (scripts/generate_inputs.py) translates these into
the one-value-per-line format required by MP-SPDZ. Students may hand-
edit the readable files and run python3 scripts/generate_inputs.py
--convert-only to regenerate the MP-SPDZ input files.

3.3 Compilation and Execution
# Start the 3 containers
docker compose up -d --build party0 party1 party2

# Compile an MPC program (replace 'my-auction' with your program name)
docker compose exec party0 bash -c \

'cd /mp-spdz && python3 compile.py my-auction'

# Run with MASCOT (one terminal per party, or use the provided scripts)
docker compose exec party0 bash -c \

'cd /mp-spdz && ./mascot-party.x -N 3 -p 0 -ip Config/IPs my-auction'
Helper scripts are provided in scripts/ (see test_run.sh, run_auction.sh,
benchmark.sh).

secrets.

for output (see Milestone 1, item 3).

asset) where you can compute the clearing price, volume and fills by hand.
